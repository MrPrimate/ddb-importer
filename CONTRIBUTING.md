# Contributing to DDB Importer

Thanks for the interest.
The best thing folks can help with is parsing improvements.
This is a never ending and ongoing process, with new content released all the time.
It is where most of the ongoing work is.

You do not need much coding experience.
Most of the work is massaging the data into the shape Foundry expects, especially third party subclasses, feats and spells.
The module parses maybe 50 to 80 percent of those correctly depending on how closely the source follows standard wording.
The closer to standard, the better it parses.

## How parsing works

The importer reads the raw D&D Beyond data, parses it into a Foundry document, then applies "enrichers" on top.
Enrichers are hints that tell the parser how to build or adjust the activities it generates: uses, damage, effects, save DCs, summons etc.

When something imports wrong, the fix is almost always writing or correcting an enricher, not touching the core parser.

## Where things live

- Enrichers, the per feature hints: `src/parser/enrichers`
- Config for features that need small adjustments rather than a full enricher: `src/config/dictionary/parsing/features.ts`
- The base class an enricher works against. Each enricher subclasses this and overrides getters like `type`, `activity`, `effects`, `override` and `additionalActivities` to describe what should change: `src/parser/enrichers/data/DDBEnricherData.ts`
- The factory that consumes those subclasses and matches them to features by name: `src/parser/enrichers/mixins/DDBEnricherFactoryMixin.ts`

The base class is heavily abstracted.
Common and fiddly parsing behaviour has been pulled into flags and helpers so they can be changed in one place rather than per feature.
(Or at least that is the intent).
That is why most of an enricher is just overriding a few getters.

## Dev setup

Clone your fork, then `npm install`. The useful scripts:

- `npm run dev` builds and watches, recompiling as you edit.
  `npm run dev:build` does a one off dev build, that is super fast for local testing.
- `npm run dev:link` symlinks the build into your Foundry modules folder so you can reload Foundry and import a character to see your change live. This is the real test loop for parsing work: edit enricher, reload, re-import, check the generated item.
- `npm run link` regenerates the enricher index. Run it after adding a new enricher file.
- `npm run lint` and `npm run typecheck` before you push.
- `npm run test` runs the vitest suite.

The repo is in typecript, and is an ongoing transition.
It is not yet fully typed.

## Adding an enricher

There is one gotcha: the file name must contain the class name, and each per folder `_module.ts` is auto generated by the link builder.
Do not edit `_module.ts` by hand.
Create your file, run `npm run link` (also run as part of `npm run dev:build`), and it gets wired in.
The parser then matches the enricher to a feature by that name.

The fastest way to learn the pattern is to copy an existing enricher for something similar to what you are fixing, then adjust it.

## Examples

### Simple: a damage activity

The Charger feat imports as a plain feature with no usable activity, so you cannot click it to roll the extra charge damage.
The fix is at `src/parser/enrichers/feat/Charger.ts`:

```ts
import DDBEnricherData from "../data/DDBEnricherData";

export default class Charger extends DDBEnricherData {

  get type() {
    return DDBEnricherData.ACTIVITY_TYPES.DAMAGE;
  }

  get activity(): IDDBActivityData {
    return {
      targetType: "enemy",
      data: {
        name: "Charge Damage",
        damage: {
          parts: [
            DDBEnricherData.basicDamagePart({
              number: 1,
              denomination: 8,
              types: DDBEnricherData.allDamageTypes(),
              }),
          ],
        },
      },
    };
  }

}
```

`type` tells the parser this feature should generate a Damage activity instead of nothing.
`activity` describes it: name it "Charge Damage" and give it a 1d8 damage part.
After this the imported feat has a clickable activity that rolls the damage.
That is the shape of most parsing fixes: find the feature that imports wrong, add or correct one or two getters, verify in Foundry.

### additionalActivities from existing actions

Sometimes D&D Beyond already parses a related action as its own separate action in the Actions tab, and you want to fold it onto the parent feature as an extra activity.
Use `additionalActivities` with an `action` block.
It looks the action up in the character data by name, pulls it in, and optionally renames it.

The Telekinetic feat, at `src/parser/enrichers/feat/Telekinetic.ts`, sets its own activity to NONE and attaches the shove action that DDB generated separately:

```ts
import DDBEnricherData from "../data/DDBEnricherData";

export default class Telekinetic extends DDBEnricherData {

  get activity(): IDDBActivityData {
    return {
      type: DDBEnricherData.ACTIVITY_TYPES.NONE,
    };
  }

  get additionalActivities(): IDDBAdditionalActivity[] {
    return [
      { action: { name: "Telekinetic Shove", type: "feat", rename: ["Shove"] } },
      { action: { name: "Telekinetic: Shove", type: "feat", rename: ["Shove"] } },
    ];
  }

}
```

`action.name` is the name to look up, `type` is where to look (here `feat`), and `rename` cleans up the resulting activity name.
Two entries are listed because the source has shipped the action under two different names over time, and only the one that matches will be used.

### additionalActivities built from scratch

When there is no existing action to pull in, build the activity yourself with an `init` plus `build` block.
`init` sets the name and activity type.
`build` flags tell the generator which sub parts to produce (damage, save, consumption, target, activation) and supply the values.
`overrides` acts as an `activity` getter for that activity.

Green Flame Blade, at `src/parser/enrichers/spell/GreenFlameBlade.ts`, adds two damage activities that do not exist in the raw data:

```ts
  get additionalActivities(): IDDBAdditionalActivity[] {
    return [
      {
        init: {
          name: "Secondary Target Damage",
          type: DDBEnricherData.ACTIVITY_TYPES.DAMAGE,
        },
        build: {
          generateDamage: true,
          generateConsumption: false,
          damageParts: [
            DDBEnricherData.basicDamagePart({
              bonus: "@mod",
              types: ["fire"],
              scalingMode: "whole",
              scalingFormula: "1d8",
            }),
          ],
        },
      },
      {
        init: {
          name: "Main Weapon Damage",
          type: DDBEnricherData.ACTIVITY_TYPES.DAMAGE,
        },
        build: {
          generateDamage: true,
          generateConsumption: false,
          damageParts: [
            DDBEnricherData.basicDamagePart({
              customFormula: "(ceil((@details.level+1)/6))d8",
              types: ["fire"],
              scalingMode: "none",
            }),
          ],
        },
        overrides: {
          data: {
            damage: {
              critical: {
                allow: true,
              },
            },
          },
        },
      },
    ];
  }
```

Each `build` describes one activity from nothing. `generateDamage: true` means "create a damage roll", and `damageParts` supplies it.
The `overrides` block is for properties the generator does not expose as a flag, set directly on the activity data.
This was added at a later stage of development and is typically easier to use an overrides block.

### Enchant activities

Enchantments apply changes to another item, such as a weapon.
Set `type` to ENCHANT, use `activity` to restrict what it can target, and define each enchantment in `effects` with `type: "enchant"`.

Magic Weapon, at `src/parser/enrichers/spell/MagicWeapon.ts`, applies a plus 1, 2 or 3 bonus depending on the slot level it was cast at:

```ts
import DDBEnricherData from "../data/DDBEnricherData";

export default class MagicWeapon extends DDBEnricherData {

  get type() {
    return DDBEnricherData.ACTIVITY_TYPES.ENCHANT;
  }

  get activity(): IDDBActivityData {
    return {
      data: {
        restrictions: {
          type: "weapon",
          allowMagical: false,
        },
      },
    };
  }

  get effects(): IDDBEffectHint[] {
    return [
      { bonus: "1", min: null, max: 3 },
      { bonus: "2", min: 4, max: 5 },
      { bonus: "3", min: 6, max: null },
    ].map((data) => {
      return {
        type: "enchant",
        name: `Magic Weapon +${data.bonus}`,
        magicalBonus: {
          makeMagical: true,
          bonus: data.bonus,
          nameAddition: `+${data.bonus}`,
        },
        options: {
          description: `This weapon has become a +${data.bonus} magic weapon, granting a bonus to attack and damage rolls.`,
        },
        data: {
          flags: {
            ddbimporter: {
              effectIdLevel: {
                min: data.min,
                max: data.max,
              },
            },
          },
        },
      } as IDDBEffectHint;
    });
  }

}
```

`activity.data.restrictions` controls what the enchantment can be applied to, here a non magical weapon.
Each entry in `effects` is one selectable enchantment.
`magicalBonus` makes the target magical and adds the attack and damage bonus, and the `effectIdLevel` min and max bind each option to a range of cast levels so the right bonus is offered for the slot used.

For enchantments that change more than a magical bonus, add a `changes` array of `ChangeHelper` calls to the effect, the same way regular active effects work.
See `src/parser/enrichers/spell/ElementalWeapon.ts` and `src/parser/enrichers/spell/Shillelagh.ts` for that.


### Generating uses from a feature

Many features have limited uses that recover on a rest.
Rather than hardcoding how many uses are spent, read it back from the imported character data with `this._getUsesWithSpent(...)` and return it from the `override` getter as
`uses`.
This keeps the current spent count in sync with what the player had on D&D Beyond, while you control the max and recovery.

Dwarf Stonecunning, at `src/parser/enrichers/trait/dwarf/Stonecunning.ts`, gives proficiency bonus uses per long rest:

```ts
  get override(): IDDBOverrideData {
    return {
      uses: this._getUsesWithSpent({
        type: "race",
        name: "Stonecunning (Tremorsense)",
        max: "@prof",
        period: "lr",
      }),
    };
  }
```

The arguments:

- `type` is where to look in the character's actions: `class`, `race` or `feat`.
- `name` is the action name to match in the DDB data. The helper finds that action and reads how many uses are already spent, so the imported item starts with the correct remaining uses.
- `max` is the maximum, as a number string like `"2"` or a formula like `"@prof"` or `"@scale.monk.focus-points"`.
- `period` is the recovery period, usually `"lr"` for long rest or `"sr"` for short rest. For non standard recovery pass a `formula` instead.
The helper returns a full uses object (spent, max and recovery), so it drops straight into `override.uses`.

Two common variations:

- Conditional max. Metamagic Adept (`src/parser/enrichers/feat/MetamagicAdept.ts`) checks whether the character already has Font of Magic and only generates uses when they do not, so a Sorcerer does not get a duplicate pool.
- Class resources. Monk's Focus (`src/parser/enrichers/class/monk/MonksFocus.ts`) uses `type: "class"`, `name: "Focus Points"`, a scale value for `max` and `period: "sr"` to track the focus point pool shared across several activities.

## Common patterns and gotchas

These come up constantly once you start writing enrichers.

### 2014 vs 2024 rules

Many features were rewritten between the 2014 and 2024 rulesets, and the same feature can need different parsing for each.
Branch on `this.is2014` and `this.is2024`:

```ts
  get effects(): IDDBEffectHint[] {
    if (this.is2014) {
      return [ /* 2014 version */ ];
    } else {
      return [ /* 2024 version */ ];
    }
  }
```

See `src/parser/enrichers/spell/Shillelagh.ts` and `src/parser/enrichers/feat/Poisoner.ts` for real examples that produce different activities per ruleset.
Spell UUID lookups also respect this: the factory resolves to 2024 spells when `this.is2024` is set.

### Adding a spell as a Cast activity

To make a feature cast a specific spell, set the activity `type` to CAST and give it `addSpellUuid` with the spell name.
The factory resolves that name to the correct compendium UUID, picking the 2014 or 2024 version automatically.
You do not hardcode the UUID.

Bard Channeler, at `src/parser/enrichers/class/bard/Channeler.ts`:

```ts
export default class Channeler extends DDBEnricherData {

  get type() {
    return DDBEnricherData.ACTIVITY_TYPES.CAST;
  }

  get activity(): IDDBActivityData {
    return {
      addSpellUuid: "Guidance",
      data: {
        spell: {
          spellbook: true,
        },
      },
    };
  }
}
```

You then add that spell to the [list here](https://github.com/MrPrimate/ddb-importer/blob/a260fc367b5a0ea5703fb8c739293076530c9500/src/config/dictionary/parsing/features.ts#L576) to prevent it been generated in the spell list and as a cast activity.

### Action vs passive feature

A feature can arrive as the feature item itself and, separately, as an action on the character.
`this.isAction` tells you which one is being parsed.
We often have to guard against this, as an enricher will be used for both the action and feature generation, often resulting in duplicate features.
If you need to guard, typically generate on the action, as this tends to have more useful metadata attached.
A very common guard is to only build an activity for the action and leave the passive feature empty:

```ts
  get type() {
    if (!this.isAction) return DDBEnricherData.ACTIVITY_TYPES.NONE;
    return DDBEnricherData.ACTIVITY_TYPES.SAVE;
  }
```

See `src/parser/enrichers/trait/dragonborn/BreathWeapon.ts`.

### Clearing auto generated effects

The parser auto generates some effects from the feature data.
When you supply your own and the automatic ones get in the way, return `true` from `clearAutoEffects` to drop them.
A common conditional form only clears them on the action copy:

```ts
  get clearAutoEffects() {
    return !this.isAction;
  }
```

### Keeping the activities the parser already built

By default returning `additionalActivities` adds to what the parser generated.
If you find your additions are replacing the default activity instead of adding to it, set `useDefaultAdditionalActivities` to `true` so the parser keeps its own activities and layers yours on top.
This is used widely, for example in `BreathWeapon` above.

### Changing what kind of item something parses as

If an item imports as the wrong document type (for example a consumable that should be a tool), override `documentStub` to set the document and system type.
See `src/parser/enrichers/item/Yarting.ts` (parsed as a tool) and `src/parser/enrichers/item/ParalysisPistol.ts` (parsed as a consumable).

### Summons

Features that summon creatures override `generateSummons` to return `true` and provide a `summonsFunction`.
There are around 40 of these, so if you are working on a summoning feature, copy the closest existing one rather than starting from scratch.

## Submitting a PR

Keep one feature or fix per PR so review stays easy. Run `npm run lint` and `npm run test` first.
In the description, say which character or content you imported to verify it, and screenshots of the item before and after in Foundry help a lot.

If you are not sure where to start, reach out on discord, and we can point you at a specific example to copy.
